# EstaCoda v0.6 Execution Plan: Memory Trust Layer + Code Structure Awareness

**Document Type:** Implementation-ready execution plan  
**Date:** 2026-05-03  
**Repo State:** `v0.5-2-g128d2aa` (doc corrections applied)  
**Source of Truth:** Actual codebase, not roadmap aspirations  
**Governance Model:** Two sequential gated tracks. Track A must pass its gate before Track B begins.

---

## 1. Executive Summary

v0.6 builds two systems with depth, not breadth:

- **Track A — Memory Trust Layer:** Make every memory entry explainable, inspectable, and selectively rendered. Provenance links tie memories to the trajectory events that created them. Deactivated memories disappear from context. Safety files (`SOUL.md`, `AGENTS.md`) remain immutable and always-rendered.
- **Track B — Code Structure Awareness Layer:** Parse the project's static import/export graph so the agent can answer "what depends on what?" without guessing from file listings.

**Track A is a hard gate for Track B.** If typecheck, smoke, or evals fail on Track A, Track B does not start.

## 3. Why v0.6 Matters

v0.4 made execution modular. v0.5 made execution observable. **v0.6 makes execution contextual and trustworthy.**

Without Track A:
- The agent cannot explain *why* it remembers something.
- Memory grows linearly and poisons context with stale, unreviewed entries.
- Users cannot inspect or correct what the agent has learned.
- Safety instructions (`SOUL.md`, `AGENTS.md`) could be accidentally deleted.

Without Track B:
- The agent has no structural understanding of the project.
- It cannot answer "what breaks if I change this file?"
- It treats the codebase as a flat file listing, not a dependency graph.

Track A is prerequisite for Track B because governed context should exist before adding structural context. The dependency graph persists in its own cache at `.estacoda/code-dependency-graph.json`, not in user/project memory.

## 4. Track A — Memory Trust Layer (Hard Gate)

### 4.1 Objective

Every memory entry carries provenance. Users and agents can inspect, deactivate, and verify memories. Rendering is selective and safe. Deactivated memories do not leak into context.

### 4.2 Hard Requirements

#### R1: Provenance metadata on promotion records

- `MemoryConclusion` gains:
  - `sourceTrajectoryId?: string`
  - `sourceEventId?: string`
  - `createdAt?: string` (ISO timestamp)
- `MemoryPromotionRecord` gains:
  - `sourceTrajectoryId?: string`
  - `sourceEventId?: string`
  - `createdAt?: string`
  - `updatedAt?: string` (already exists, verify populated)
- When `memoryProvider.conclude()` is called, the caller must supply `sourceTrajectoryId` and `sourceEventId` from the active trajectory.
- `RunRecorder` or `AgentLoop` must capture these IDs at promotion time.

#### R2: Memory promotion event in trajectory

- When a memory is promoted, record a `memory-promotion` trajectory event with:
  - `conclusionId`
  - `kind`
  - `targetFile`
  - `sourceTrajectoryId`
  - `sourceEventId`

#### R3: CLI command surface

Create `src/cli/knowledge-commands.ts` implementing memory subcommands:

```bash
estacoda knowledge memory list [--active-only] [--kind preference|fact] [--limit N]
estacoda knowledge memory inspect <id>
estacoda knowledge memory deactivate <id>
```

- `list` shows: id, kind, active status, content (truncated), createdAt.
- `inspect` shows: full record, provenance IDs, source sessions, confidence, occurrences.
- `deactivate` marks the promotion record `active: false`. Physical file removal happens only when there is a stable anchor or exact safe match. Deactivation must suppress rendered context.

**No hard delete.** Deactivation preserves audit trail.

#### R4: Agent-facing memory tools

Add two tools:

| Tool | Risk Class | Behavior |
|------|-----------|----------|
| `knowledge.memory.inspect` | `read` | Lists or searches memory entries. Low risk. |
| `knowledge.memory.deactivate` | `workspace-write` | Deactivates a memory entry by ID. Medium risk, requires approval. |

`knowledge.memory.deactivate` must:
- Reject IDs for entries in `SOUL.md` or `AGENTS.md` with clear error.
- Reject unknown IDs.
- Require explicit user approval before deactivating.

#### R5: Safety file protection

- `SOUL.md` and `AGENTS.md` are **always rendered in full** regardless of selective rendering settings.
- `SOUL.md` and `AGENTS.md` entries **cannot be deactivated** through `knowledge.memory.deactivate` or CLI.
- This is enforced in `MemoryInspector.deactivate()` and the `knowledge.memory.deactivate` tool handler.

#### R6: Legacy memory compatibility

- Old entries without provenance are marked `provenance: "legacy"` or `provenance: "unknown"`.
- New v0.6 promotions must have non-null `sourceTrajectoryId` and `sourceEventId`.
- Old entries remain inspectable and renderable under compatibility behavior (treated as always-active unless explicitly deactivated).
- The selective renderer must handle both legacy and provenance-aware entries without error.

#### R7: Selective rendering for USER.md / MEMORY.md

Create `src/memory/selective-renderer.ts` with the following behavior:

**Input:** `MemorySnapshot` + `RenderContext` (optional query string, recency bias)

**Policy:**
1. Start with all active entries from `USER.md` and `MEMORY.md`.
2. If `query` is provided:
   - Rank entries by relevance: exact substring match > partial match > no match.
   - Include only entries with relevance score > 0 (some match).
   - If **no entries match**, return the **N most recent entries** (where N = 3 default, configurable).
   - **Never fall back to full dump.**
3. If `query` is not provided:
   - Return all active entries (backward-compatible behavior).
4. Always include full `SOUL.md` and `AGENTS.md`.
5. Include `omittedCount` and `renderMode: "selective" | "full"` in output metadata.

**Relevance scoring (v0.6):**
- Exact substring of query in entry content: score = 1.0
- Any word from query in entry content: score = 0.5
- No match: score = 0.0

This is intentionally simple. Semantic relevance is deferred.

#### R8: Render integration

- `LocalMemoryProvider.context()` gains optional overload: `context(options?: { query?: string })`.
- Default behavior (no query) = full render (backward compatible).
- When query is provided = selective render.
- The query is passed from the intent router or from the agent's own context request.

#### R9: Regression fixtures

Create eval fixtures in `src/eval/fixtures/`:

| Fixture | What It Tests |
|---------|---------------|
| `memory-promotion-provenance.ts` | Trigger promotion, assert `sourceTrajectoryId` and `sourceEventId` are non-null. |
| `memory-deactivate-suppresses.ts` | Activate promotion, deactivate it, assert it does not appear in rendered context. |
| `memory-selective-renders.ts` | Create snapshot with 10 entries, query for 1 specific entry, assert only relevant + recent fallback returned. |
| `memory-safety-files-protected.ts` | Attempt to deactivate SOUL.md/AGENTS.md entries, assert rejection. |

## 5. Track B — Code Structure Awareness Layer (Gated)

### 5.1 Objective

Parse the project's static import/export relationships so the agent can answer dependency questions without executing code or guessing from filenames.

### 5.2 Hard Requirements

#### R1: CodeDependencyGraph component

Create `src/knowledge/code-dependency-graph.ts`:

```typescript
export type ModuleNode = {
  id: string;           // relative file path
  kind: "ts" | "js" | "json" | "external";
  sizeBytes: number;
};

export type ImportEdge = {
  from: string;         // importer module id
  to: string;           // imported module id
  kind: "static" | "dynamic" | "type-only";
  sourceLine: number;
};

export type CodeDependencyGraph = {
  nodes: ModuleNode[];
  edges: ImportEdge[];
  generatedAt: string;
};
```

**Parser scope:**
- TypeScript and JavaScript files in the project workspace.
- Regex-based parsing (no TypeScript compiler API dependency).
- Handles:
  - `import X from '...'`
  - `import { X } from '...'`
  - `import * as X from '...'`
  - `import type { X } from '...'`
  - `export { X } from '...'`
  - `export * from '...'`
  - `require('...')`
  - `import('...')` and `await import('...')` (marked as dynamic)
- Resolves relative paths to absolute, then back to project-relative.
- **Ignores `node_modules` imports** (do not resolve).
- **Ignores external package imports** (record as `to: "<external:package-name>"`, node `kind: "external"`).
- **Ignore rules:** Skip `node_modules`, `dist`, `build`, `.git`, `.estacoda`, `coverage`. Skip `*.d.ts` for v0.6 unless explicitly needed.

#### R2: Forward dependency lookup

```typescript
function forwardDeps(graph: CodeDependencyGraph, moduleId: string): string[];
```

Returns the list of module IDs this file directly imports.

#### R3: Reverse dependency lookup

```typescript
function reverseDeps(graph: CodeDependencyGraph, moduleId: string): string[];
```

Returns the list of module IDs that directly import this file.

#### R4: Affected-file lookup (simple transitive)

```typescript
function affectedFiles(graph: CodeDependencyGraph, moduleId: string, maxDepth?: number): string[];
```

Returns all module IDs that transitively depend on this file (reverse closure).

- `maxDepth` defaults to 5 to prevent runaway traversal.
- Simple BFS/DFS over reverse edges.
- No semantic analysis (does not check if the changed export is actually used).

#### R5: Graph summary

```typescript
function graphSummary(graph: CodeDependencyGraph): {
  nodeCount: number;
  edgeCount: number;
  isolatedFiles: string[];       // nodes with zero in-degree and out-degree
  highlyConnected: string[];     // nodes with in-degree + out-degree > threshold
  externalDependencies: string[]; // unique external package names
};
```

#### R6: Caching

- Cache graph at `<project>/.estacoda/code-dependency-graph.json`.
- Store `generatedAt` and a hash of scanned file mtimes.
- On query, check if cache is stale (any source file newer than `generatedAt`).
- Regenerate lazily on first stale query.

#### R7: CLI / tool access

Add `estacoda knowledge code` CLI commands:

```bash
estacoda knowledge code deps <file-path>          # forward deps
estacoda knowledge code rdeps <file-path>         # reverse deps
estacoda knowledge code affected <file-path>      # transitive affected files
estacoda knowledge code summary                   # graph summary
estacoda knowledge code refresh                   # force regeneration
```

Add agent tool:

| Tool | Risk Class | Behavior |
|------|-----------|----------|
| `knowledge.code.query` | `read` | Query dependency graph. Low risk. |

Tool input:
```json
{
  "moduleId": "src/memory/memory-store.ts",
  "query": "forward" | "reverse" | "affected" | "summary"
}
```

#### R8: Deterministic fixtures

Create eval fixtures:

| Fixture | What It Tests |
|---------|---------------|
| `knowledge-forward-deps.ts` | Parse temp project, assert forward deps match expected imports. |
| `knowledge-reverse-deps.ts` | Parse temp project, assert reverse deps match expected importers. |
| `knowledge-affected-files.ts` | Parse temp project with chain A→B→C, assert affected(A) includes B and C. |
| `knowledge-graph-summary.ts` | Parse temp project, assert summary counts match expected. |
| `knowledge-cache-invalidates.ts` | Generate graph, modify source file, assert next query regenerates. |

## 7. Non-Goals (Strict Exclusions)

1. **Semantic memory search** (embeddings, vector DB) — out of MVP scope.
2. **Memory compression/summarization** — requires LLM calls; too complex.
3. **Automatic memory policy evolution** — v0.7 curator responsibility.
4. **Multi-language dependency parsing** — TypeScript/JavaScript only.
5. **node_modules resolution** — external packages recorded as `<external:name>` only.
6. **Semantic change-impact analysis** — affected-files is transitive graph only, not usage-based.
7. **Knowledge graph visualization** — no UI, no Mermaid generation from live data.
8. **Cross-project memory federation** — single project scope only.
9. **Vitest / test refactoring** — deferred to post-v0.6.
10. **TaskFlow / durable workflows** — v0.8.
11. **Skill evolution pipeline activation** — v0.7.
12. **General "knowledge graph" beyond code dependencies** — no ontology, no semantic web, no reasoning engine.

## 9. Security & Trust Implications

| Concern | Mitigation |
|---------|------------|
| `knowledge.memory.deactivate` removes security memories | Reject deactivation of `SOUL.md` / `AGENTS.md` entries. Require approval for all others. |
| Provenance links leak trajectory content | Store only IDs, not content, in provenance fields. |
| Selective renderer hides critical instructions | `SOUL.md` / `AGENTS.md` always rendered in full. Selective filter applies only to `USER.md` / `MEMORY.md`. |
| Dependency graph parser executes code | Static regex parsing only. No `require()`, `eval()`, or dynamic execution. |
| Dependency graph exposes project structure | Read-only. No write access. Only operates on trusted workspace files. |
| Cache file tampering | Cache is advisory; stale cache is detected via mtime check and regenerated. |

## 10. Failure Modes

| Failure | Manifestation | Detection |
|---------|--------------|-----------|
| Provenance fields are null | `inspect` shows empty source fields | Eval fixture `memory-promotion-provenance` |
| Deactivated memory still renders | Context includes "forgotten" preference | Eval fixture `memory-deactivate-suppresses` |
| Selective renderer dumps everything | `omittedCount: 0` when query is specific | Eval fixture `memory-selective-renders` |
| Safety file deleted | `SOUL.md` or `AGENTS.md` missing from context | Eval fixture `memory-safety-files-protected` |
| Dependency graph misses imports | Forward deps list is empty for file with imports | Eval fixture `knowledge-forward-deps` |
| Cache never invalidates | Stale graph returned after file change | Eval fixture `knowledge-cache-invalidates` |
| Affected-files too shallow | Transitive dependents missed | Eval fixture `knowledge-affected-files` |
| Smoke monolith hits TS limits | `typecheck` fails with TS2563 | `bun run typecheck` |

## 12. Open Questions (Blocking Decisions)

1. **Q: Should `sourceEventId` reference a trajectory event or session event?**
   - **Recommendation:** Trajectory event IDs. They are v0.5's canonical record.
   - **Status:** Not blocking if we agree.

2. **Q: How many "recent fallback" entries when query matches nothing?**
   - **Recommendation:** 3 entries. Configurable via `RenderOptions`.
   - **Status:** Not blocking if we agree.

3. **Q: Should dependency graph use TypeScript compiler API or regex?**
   - **Recommendation:** Regex for v0.6. No new dependency. Document that type-only imports and complex re-exports may be missed.
   - **Status:** Not blocking if we agree.

4. **Q: Should memory metadata be inline in markdown or stored separately?**
   - **Recommendation:** Stored separately in `MemoryPromotionStore`. Keep markdown clean and human-editable.
   - **Status:** Not blocking if we agree.
