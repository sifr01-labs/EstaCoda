# v0.5 Documentation and Roadmap Alignment Audit

**Date:** 2025-05-03
**Repo state:** v0.5 tag at `b8bfe68`
**Auditor:** Agent review of docs, code, and roadmap narrative
**Source of truth:** Actual codebase (`git ls-files`, `wc -l`, runtime behavior)

---

## 1. Executive Summary

v0.5 delivered the evidence substrate as planned. The codebase is materially ahead of several docs. The biggest risk is **doc drift**: multiple files still describe v0.3-era limitations as current reality, and some files describe planned behavior as if implemented.

**Verdict:** We are on track. v0.5 slightly over-delivered on eval infrastructure (golden flows, change manifests) while remaining within scope. The primary action before v0.6 is **documentation correction and consolidation**, not code work.

| Roadmap Claim | Delivered? | Notes |

| Structured persisted record per run | ✅ Yes | `trajectories` table in SQLite |
| Tool calls timestamped and tied to context | ✅ Yes | Each event has `timestamp` and `data` |

### Did v0.5 Overreach?

**Slightly, in a good way.**

The execution plan added:
- Golden flow schema and comparison logic (roadmap mentions golden flows as v0.5 stretch)
- `ChangeManifestStore` with full CRUD + evidence linkage (roadmap frames this as v0.7)
- Redaction engine (not explicitly in roadmap but necessary for safe trace export)

These are bounded additions that fit the v0.5 theme (evidence substrate) and do not create v0.6 dependencies.

### Did v0.5 Underdeliver?

**No.** All must-have blockers are complete. The two deferred stretch items are:
1. `estacoda trace compare` — low value, can be done with `dump` + external diff
2. Retroactive failure classification on old sessions — old sessions predate classification; not critical

## 9. Risks Before v0.6

### High

1. **Doc drift causing wrong engineering decisions.** If an incoming agent reads `docs/architecture/overview.md` and believes trajectories are "in-memory only," they may reimplement persistence or waste time.
2. **Handoff guide describes v0.3 as current.** New agents will start with wrong priorities.

### Medium

3. **Smoke test monolith continues growing.** At 14K lines, it may hit further TypeScript limits or become unmaintainable. The `// @ts-nocheck` addition in v0.5 is a warning sign.
4. **Two roadmap files (`ROADMAP.md` and `docs/ROADMAP.md`).** They have different content. This creates confusion about which is authoritative.
5. **No `CONTRIBUTING.md` despite AGENTS.md referencing it.** External contributors have no entry point.

### Low

6. **Golden flows are static JSON, not generated from real runs.** This is by design for determinism, but the gap between "recorded successful run" and "manually authored JSON" should be documented.
7. **Redaction is heuristic, not cryptographic.** Safe for MVP but should be noted in security docs.

## 10. Recommended v0.6 Planning Direction

The roadmap narrative defines v0.6 as:

> "Memory, Dependency Graph, and Knowledge Graph. Make learning useful without poisoning context."

### What v0.6 Should Focus On

1. **Memory provenance and inspection**
   - Link memory promotions to trajectory events
   - Add `memory inspect` CLI or tool action
   - Add memory deletion flow

2. **Project knowledge graph**
   - Code dependency graph integration (e.g., parse imports/exports)
   - Make graph output available to agent planning

3. **Memory rendering improvements**
   - Selective rendering (not dump-based)
   - Freshness/staleness handling
   - Eval fixtures for memory rendering behavior

4. **Scope separation**
   - Explicitly separate user-level memory from project-level memory
   - Ensure `USER.md` vs `MEMORY.md` boundaries are enforced

### What v0.6 Should NOT Do

- Do NOT make memory fully autonomous by default
- Do NOT store sensitive memories without explicit approval
- Do NOT build a bloated knowledge graph visualization
- Do NOT start TaskFlow (that is v0.8)
- Do NOT start skill evolution pipeline (that is v0.7)

### Sequencing Note

v0.6 can start immediately. There are no v0.5 blockers. The doc corrections from this audit should be done in parallel with v0.6 planning, not as a prerequisite.
