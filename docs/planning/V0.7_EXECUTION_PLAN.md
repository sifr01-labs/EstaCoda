2. **Decomposed agent loop**: `RunRecorder`, `ToolPlanRunner`, `ProviderTurnLoop`, `SkillWorkflowExecutor`, `NativeToolExecutor` are separate. Clean insertion points for telemetry.
3. **Trajectory schema** (32 event kinds): `memory-promotion`, `skill-route`, `skill-used`, `skill-outcome`, `tool-execution`, `security-escalation`, `prompt-assembly`, etc.
4. **Memory provenance**: `sourceTrajectoryId`, `sourceEventId` on all new promotions.
5. **Failure classification**: 13 coarse classes persisted to SQLite.
6. **Bundled skill sync**: Hash-based manifest with user-modified guard.
7. **Skill mutation policy**: Authority expansion detection, pinning, path safety.

# Phase 4 — v0.7 Execution Plan

## 4.1 Objective

Governed skill evolution and curator infrastructure that converts trace evidence, failures, user corrections, and eval outcomes into reviewable improvement proposals.

## 4.2 Product Thesis

> Self-improvement is not silent mutation. It is evidence-backed, eval-tested, reviewable, reversible harness improvement.

## 4.3 Workstreams

### Track A — Evidence Model & User Corrections (Hard Gate)

**Goal**: Close the data-model gaps that prevent automated proposal generation.

#### A1. User Correction Capture

Add a structured `user-correction` trajectory event kind and wire it into the runtime.

- When the user says "no, do it this way" or "that's wrong, fix X", record it.
- Link corrections to the skill/tool that produced the incorrect behavior.
- Store corrections in `SkillEvolutionStore` as a new JSONL stream: `corrections.jsonl`.

**Files**: `src/contracts/trajectory.ts`, `src/trajectory/trajectory-recorder.ts`, `src/skills/skill-evolution.ts`, `src/runtime/agent-loop.ts`

#### A2. Observation-to-Manifest Bridge

When `skill.observe` is called with a `candidateImprovement`, automatically create a lightweight `EvolutionChangeManifest` via `ChangeManifestStore`.

- Link the observation ID to the manifest evidence.
- Set status to `proposed`.
- Populate hypothesis from the observation `lesson`.
- Set risk level based on `sourceTrust` and `requiresHumanApproval`.

**Files**: `src/skills/skill-evolution.ts`, `src/skills/change-manifest-store.ts`

#### A3. Proposal-to-Manifest Bridge

When `skill.propose_patch` is called, create or link a `ChangeManifest`.

- Populate `filesChanged` with the skill path.
- Link observation IDs to `evidence.observations`.
- Link trajectory IDs from the current run to `evidence.traces`.
- Populate `hypothesis`, `predictedImpact`, `riskLevel`, `evalCommand`, `constraintGates`, `rollbackPlan`.

**Files**: `src/skills/skill-tools.ts`, `src/skills/skill-evolution.ts`, `src/skills/change-manifest-store.ts`

#### A4. Eval Gate Wiring

When `skill.promote_patch` runs eval gates, record the result in both `SkillEvalRunRecord` and `EvolutionChangeManifest`.

- If eval gates fail, block promotion and update manifest status to `rejected`.
- If eval gates pass, allow promotion and update manifest status to `promoted`.

**Files**: `src/skills/skill-tools.ts`, `src/skills/skill-evolution.ts`

### Track B — Curator CLI & Review Workflow (Gated)

**Goal**: Give users a CLI surface to inspect, review, approve, reject, and promote skill improvements.

#### B1. Skill CLI Namespace

Create `src/cli/skill-commands.ts`:

```bash
estacoda skill list [--source official|personal|project|external] [--active-only]
estacoda skill inspect <name>
estacoda skill usage <name>
estacoda skill eval <name> [--dry-run]
```

**Files**: `src/cli/skill-commands.ts`, `src/cli/cli.ts`

#### B2. Curator CLI

```bash
estacoda skill curator status [--skill <name>]
estacoda skill curator archive <name>
estacoda skill curator restore <name>
estacoda skill curator pin <name>
estacoda skill curator unpin <name>
```

- `status` shows `summarizeSkillCuratorStatus` output.
- `archive` sets skill state to `archived`.
- `restore` sets state back to `active`.
- `pin`/`unpin` toggles pinned flag.

**Files**: `src/cli/skill-commands.ts`, `src/skills/skill-curator-status.ts`, `src/skills/skill-registry.ts`

#### B3. Proposal CLI

```bash
estacoda skill proposals list [--skill <name>] [--status proposed|approved|rejected|promoted]
estacoda skill proposals inspect <id>
estacoda skill proposals approve <id>
estacoda skill proposals reject <id> [--reason <text>]
estacoda skill proposals promote <id> [--dry-run]
```

**Files**: `src/cli/skill-commands.ts`, `src/skills/skill-evolution.ts`, `src/skills/skill-tools.ts`

#### B4. Manifest CLI

```bash
estacoda manifest list [--target skill|tool|prompt|policy] [--status ...]
estacoda manifest inspect <id>
estacoda manifest approve <id>
estacoda manifest reject <id> [--reason <text>]
estacoda manifest link-evidence <id> --trace <trace-id>
```

**Files**: `src/cli/manifest-commands.ts`, `src/skills/change-manifest-store.ts`, `src/cli/cli.ts`

#### B5. In-Session Slash Commands

Add slash commands for skill review:

```
/skills curator
/skills proposals
/skill inspect <name>
/skill approve <proposal-id>
/skill reject <proposal-id>
```

**Files**: `src/cli/slash-menu.ts`, `src/cli/session-loop.ts`

### Track C — Tool-Description Proposals & Export Skeleton (Gated)

**Goal**: Represent tool-description and routing-metadata proposals as target types, and add a clean export format for future optimization.

#### C1. Tool-Description Proposal Skeleton

Extend `skill.propose_patch` (or create `tool.propose_description`) to support `target: "tool_description"`.

- Store proposed tool description changes as `EvolutionChangeManifest` with target `"tool_description"`.
- Do NOT implement auto-application. Just represent the proposal type.
- Add eval fixture: tool-description proposal can be created and inspected.

**Files**: `src/tools/tool-registry.ts`, `src/contracts/evolution.ts`, `src/skills/change-manifest-store.ts`

#### C2. Routing-Metadata Proposal Skeleton

Similarly represent `target: "system_prompt_section"` for routing metadata.

- Store proposed routing hints as manifests.
- No auto-application.
- Eval fixture only.

**Files**: `src/runtime/intent-router.ts`, `src/contracts/evolution.ts`

#### C3. DSPy/GEPA-Compatible Export

Create `src/evolution/export-format.ts` with the dataset type defined in Phase 3.4.

Add CLI:

```bash
estacoda evolution export --dataset <path> [--since <date>] [--skill <name>]
```

- Exports traces, eval runs, observations, and proposals as clean JSON.
- No Python dependency.
- Document the format for future DSPy/GEPA consumption.

**Files**: `src/evolution/export-format.ts`, `src/cli/evolution-commands.ts`, `src/cli/cli.ts`

## 4.4 Non-Goals (Strict Exclusions)

1. **No full autonomous runtime self-modification** — proposals must be reviewed before promotion.
2. **No direct code evolution** — `runtime_code` target type exists in schema but is excluded from v0.7.
3. **No silent skill rewriting** — all promotions require explicit approval or configured policy.
4. **No full DSPy/GEPA integration** — only a clean JSON export skeleton.
5. **No Python optimization pipeline** — deferred until post-MVP.
6. **No marketplace** — deferred to v0.10+.
7. **No enforced workflow for every skill** — advisory mode remains default.
8. **No bypassing security/approval layers** — all mutations go through existing gates.
9. **No broad semantic architecture inference** — code dependency graph is regex-based only.
10. **No automatic memory policy evolution** — memory remains manually curated in v0.7.
11. **No embedding/vector search** — out of MVP scope.

## 4.8 Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Fake self-evolution (silent mutation) | Low | Critical | Require explicit approval for all promotions; no auto-promotion in v0.7 |
| Weak evidence quality | Medium | Medium | Require minimum observation count before proposal; validate hypothesis length |
| Proposal spam | Medium | Medium | Rate-limit proposals per skill; require evidence thresholds |
| Memory poisoning feeding skill patches | Medium | High | Memory promotion already has provenance; do not auto-promote memory-based proposals |
| Eval overfitting | Low | Medium | Eval fixtures must be deterministic and diverse; document limits |
| Unsafe third-party skill mutation | Low | Critical | External skills are read-only; bundled skills require rebaseline |
| Silent drift | Low | Critical | All changes go through manifest store with audit trail |
| Too much abstraction too early | Medium | Medium | Keep tool-description and routing-metadata as skeleton only; no auto-application |
| Premature DSPy/GEPA dependency | Low | Medium | Export is JSON-only; no Python pipeline |
| Breaking existing skills | Medium | High | Comprehensive smoke coverage; rollback tool exists; eval gates block bad promotions |
| CLI complexity explosion | Medium | Medium | Group commands under namespaces (`skill`, `manifest`); keep slash commands minimal |
| create-runtime.ts god factory bloat | High | Medium | Do not add more constructor calls; extract evolution orchestrator into its own module |

## 4.12 Open Questions

1. **User correction detection**: Should the agent auto-detect corrections from user text ("no, that's wrong"), or should corrections be explicit via a tool/command?
   - **Recommended**: Start with explicit `/correct` slash command or `skill.observe` with type `user_correction`. Auto-detection is v0.8+.

2. **Auto-proposal generation threshold**: How many observations before auto-generating a proposal?
   - **Recommended**: Minimum 3 observations of the same skill with consistent `candidateImprovement`. Configurable in settings.

3. **Curator automation level**: Should curator ever auto-archive or auto-restore without user approval?
   - **Recommended**: No. Curator recommends; user or configured policy decides. Default is manual review.

4. **Eval command in manifest**: Should `evalCommand` be a shell command string or a structured object?
   - **Recommended**: Shell command string for flexibility, but validate it starts with `bun run` or `estacoda eval`.

5. **Manifest CLI vs Proposal CLI**: Should proposals and manifests share a CLI namespace or be separate?
   - **Recommended**: Separate. `estacoda skill proposals` for skill patches; `estacoda manifest` for all evolution manifests (including future tool/prompt types).

# Audit Summary (Terminal/Chat)

## What Exists (Strong Foundation)

- **SkillEvolutionStore** (666 lines): Full usage tracking, observations, patch proposals, promotions, eval runs. JSONL persistence.
- **SkillLearningManager** (497 lines): Auto-creates local skills from repeated workflows.
- **SkillTools** (2,292 lines): 22 agent tools including full proposal lifecycle (propose, list, review, approve, reject, promote).
- **ChangeManifestStore** (137 lines): Full `EvolutionChangeManifest` schema with propose/list/find/update/linkEvidence.
- **SkillCuratorStatus** (100 lines): Pure recommendation function.
- **Trajectory** (32 event kinds): SQLite persistence, CLI inspection, redaction.
- **Failure classification** (13 classes): Heuristic, persisted.
- **Eval** (13 fixtures): Deterministic regression suite.
- **Security**: Hard floor, policy modes, workspace approvals/trust.
- **Modular smoke harness**: Fresh factories, filtering, structured reporting.

## What's Missing (v0.7 Work)

- **ChangeManifestStore is orphaned** — not wired into runtime, tools, or CLI.
- **Curator does not act** — pure function, no CLI, no automation.
- **No skill/curator/proposal CLI** — only agent tools exist.
- **No user correction capture** — structured event type missing.
- **Skill evals are metadata-only** — no real fixture execution.
- **No auto-proposal generation** — evidence exists, no pipeline.
- **No DSPy/GEPA export** — no clean dataset export format.
- **Tool-description/routing-metadata proposals are schema-only** — no implementation.

## What Surprised the Audit

The skills subsystem is far more mature than the roadmap suggests. The gap is not in building new infrastructure but in **activating** existing infrastructure. The evolution store, proposal types, mutation policy, bundled sync, and rollback all exist. What's missing is the glue: manifest wiring, CLI surface, and the automation loop.

## Risk Assessment

**Low risk for v0.7**: The substrate is strong. No architectural blockers.
**Medium risk**: `create-runtime.ts` god factory (901 lines). Adding evolution orchestration could bloat it further. Extract an `EvolutionOrchestrator` class early.
**High risk**: Fake self-evolution. Mitigation: explicit approval gates, no auto-promotion, comprehensive smoke coverage.

## Recommended First Task

**A2: Observation-to-Manifest Bridge**. This is the smallest change with the highest impact. It activates the orphaned `ChangeManifestStore` and makes the entire proposal system reviewable through manifests.
