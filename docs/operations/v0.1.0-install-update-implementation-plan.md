## Install Ownership Modes

EstaCoda distinguishes six install ownership modes. The update command routes behavior based on the detected mode.

1. **managed-source**
   - Created by curl | bash installer into an installer-owned directory such as `~/.estacoda/estacoda` or `/usr/local/lib/estacoda`.
   - EstaCoda may run guarded source updates here.

2. **manual-source**
   - Created by a user through `git clone` + `setup-estacoda.sh`.
   - EstaCoda must treat this as a contributor/manual checkout by default.
   - `estacoda update` may check and advise, but must not assume ownership of the repo.

3. **homebrew**
   - Managed by Homebrew.
   - `estacoda update` routes to `brew upgrade`.

4. **docker**
   - Managed by container runtime.
   - `estacoda update` routes to `docker pull` / `compose pull`.

5. **npm-global**
   - Managed by npm.
   - `estacoda update` routes to `npm install -g estacoda@latest`.

6. **pnpm-global**
   - Managed by pnpm where supported.
   - `estacoda update` routes to `pnpm add -g estacoda@latest`.

## 1. Current-State Audit

### scripts/install.sh

**File exists.** A POSIX shell installer that probes for prebuilt release artifacts and falls back to a source build.

Current behavior:
- Checks Node.js >= 22.18.0, including Node 23+.
- Checks Corepack availability
- Probes GitHub releases for a platform artifact; 404s silently in dev
- On artifact miss: runs `pnpm install --frozen-lockfile && pnpm run build` from repo root relative to script location
- Writes a bash wrapper script to `~/.estacoda/bin/estacoda` pointing at `dist/index.js`
- Appends PATH exports to shell rc files
- Symlinks into `~/.local/bin/` if present in PATH

Limitations:
- **Not a standalone curl|bash installer.** It requires the repo to already exist (derives `REPO_ROOT` from `SCRIPT_DIR`).
- No `--branch`, `--dir`, `--skip-init` flags.
- No OS/arch validation matrix.
- No install-method stamp.
- No managed-install directory tracking.
- No root/FHS install path.
- No Termux layout detection.
- No WSL detection.

### scripts/estacoda-wrapper.sh

## 4. Safety Boundaries

These rules are mandatory for all install/update PRs:

1. **No `git reset --hard` outside explicitly managed installer directories.**
   - Only directories stamped with `managed-source` install-method metadata may be reconciled.
   - `manual-source` worktrees use check-and-advise behavior; `estacoda update` must not self-mutate unless a future explicit flag is added.
   - Contributor worktrees use `ff-only` update behavior.

2. **Dirty worktrees must not be silently destroyed.**
   - Source update must refuse to proceed if the worktree is dirty.
   - Auto-stash and explicit dirty-worktree override behavior are deferred until a later hardening PR.

3. **Managed clone directories may reconcile to origin/main only if install-method metadata proves ownership.**
   - The stamp file must contain: install method, install date, source URL, expected branch.
   - If install method is `managed-source` and stamp matches: allow guarded source update.
   - If install method is `manual-source`: check and advise; do not self-mutate unless a future explicit flag is added.
   - If the stamp is missing or mismatched, treat as manual/contributor repo.

4. **User state under `~/.estacoda` must not be destroyed.**
   - PR-I5 source update rollback is limited to the managed source checkout's pre-pull SHA.
   - Broader state backup/snapshot behavior belongs to PR-I6.
   - `applyUpdate()` must not touch `~/.estacoda/profiles/`, `~/.estacoda/memory/`, `~/.estacoda/sessions.sqlite`.

5. **Homebrew updates route to `brew upgrade`.**
   - `estacoda update` must detect Homebrew install and output:
     ```
     Homebrew install detected. Run: brew upgrade kemetresearch/tap/estacoda
     ```
   - Must not attempt self-mutation.

6. **Docker updates route to `docker pull`.**
   - `estacoda update` must detect Docker install and output:
     ```
     Docker install detected. Run: docker pull ghcr.io/kemetresearch/estacoda:latest
     ```
   - Must not attempt self-mutation.
   - If running inside a container image, install method resolves to `docker` unless an explicit stamp says otherwise.
   - `estacoda update` must not mutate the container filesystem as a source checkout.

7. **npm-global updates route to `npm install -g estacoda@latest`.**
   - `estacoda update` must detect npm-global install and output the npm command.
   - Must not attempt self-mutation.

8. **pnpm-global updates route to `pnpm add -g estacoda@latest`.**
   - `estacoda update` must detect pnpm-global install and output the pnpm command.
   - Must not attempt self-mutation.

9. **Release artifact binary update remains unsupported unless implemented and validated separately.**
   - The `ESTACODA_UPDATE_ARTIFACT` path in `update-engine.ts` may be removed or deprecated.
   - Binary-only update is not a v0.1.0 launch requirement.

## 5. Installer Design

### curl | bash installer (`scripts/install.sh`)

Intended behavior after PR-I2:

1. Download raw `scripts/install.sh` from GitHub:
   ```bash
   curl -fsSL https://raw.githubusercontent.com/KemetResearch/EstaCoda/v0.1.0/scripts/install.sh | bash
   ```
2. Detect OS/arch (macOS 11+ check, Linux glibc check)
3. Enforce Node.js >= 22.18.0
4. Ensure Corepack/pnpm
5. Default non-root install:
   - Clone into `~/.estacoda/estacoda` (managed source dir)
   - Build `dist/`
   - Symlink wrapper to `~/.local/bin/estacoda`
   - Stamp install method: `~/.estacoda/estacoda/.install-method.json` (method: `managed-source`, source URL, branch)
6. Optional flags:
   - `--branch <branch>`: clone specific branch instead of default
   - `--dir <path>`: custom install directory
   - `--skip-init`: skip `estacoda init`
   - `--fhs`: root Linux FHS install into `/usr/local/lib/estacoda` + `/usr/local/bin/estacoda`
7. Preserve existing managed install safely (rebuild rather than re-clone if stamp matches)
8. Add PATH entry to shell rc if needed

### git clone + setup script (`scripts/setup-estacoda.sh`)

Intended behavior after PR-I2:

1. User runs:
   ```bash
   git clone https://github.com/KemetResearch/EstaCoda.git
   cd EstaCoda
   ./scripts/setup-estacoda.sh
   ```
2. Verify Node.js >= 22.18.0
3. `corepack enable` + `pnpm install`
4. `pnpm run build`
5. Stamp install method in repo root: `.install-method.json` (method: `manual-source`, source URL, branch)
   - The setup script may write `.install-method.json`, but the stamp must distinguish `manual-source` from `managed-source`.
6. Offer to symlink `scripts/estacoda-wrapper.sh` to `~/.local/bin/estacoda`
7. Run `estacoda init` unless `--skip-init`

### npm install -g (`npm install -g estacoda`)

Intended behavior after final npm publication, with package metadata and tarball installability made ready by PR-I1:

1. `npm install -g estacoda` or `pnpm add -g estacoda`
2. The global package manager handles the install.
3. `estacoda update` detects `npm-global` or `pnpm-global` and routes to the appropriate package manager upgrade command.
4. No source mutation is attempted.

### npm publish workflow and version policy

PR-I1 makes the package publish-ready and validates installability from a packed tarball, but it does not publish to npm and does not bump the version.

Final release flow:

1. Keep `package.json` version unchanged through PR-I1.
2. In the final release PR only, bump `package.json` to `0.1.0`.
3. Run the full validation set, including `pnpm run verify:package-bin` and a temp-prefix global install from the packed tarball.
4. Run `npm publish --dry-run` and inspect the package contents.
5. Tag `v0.1.0`.
6. Publish with `npm publish --access public` from the clean release commit.
7. Verify `npm install -g estacoda` from a clean environment after publication.

## 6. Update Command Contract

### Flags

| Flag | Behavior |
|---|---|
| `estacoda update --check` | Check only. Never modify files. Exit 0 if update available, exit 2 if up-to-date, exit 1 on error. |
| `estacoda update` | Default safe apply. If managed-source: fetch, check ff-only safety, check worktree cleanliness, pull, install deps, build, validate. If manual-source: check and advise; do not self-mutate. If homebrew/docker/npm-global/pnpm-global: print routing message and exit 0. |
| `estacoda update --yes` | Apply without interactive confirmation where appropriate. Still refuses unsafe operations (e.g. non-ff pull, non-managed dir). |
| `estacoda update --backup` | Force backup/snapshot before applying. |
| `estacoda update --no-backup` | Skip backup if user is confident. Not recommended. |
| `estacoda update --gateway` | Gateway-aware update. Avoid interactive prompts. Write update progress to log. After successful source update, restart or instruct restart of the managed gateway service where the service manager is available. If the gateway is not service-managed, print an explicit manual restart instruction. Never restart user processes blindly. |

### Exit codes

| Code | Meaning |
|---|---|
| `0` | Success: update applied, or update available info shown, or routing message printed. |
| `1` | Error: network failure, build failure, safety refusal, backup failure. |
| `2` | Up-to-date: no action needed. |
| `3` | Dirty worktree: user must stash/commit or pass `--yes` to auto-stash. |
| `4` | Non-managed install: update refused because install-method stamp is missing or mismatched. |

### Managed install behavior

- If install method is `managed-source` and stamp matches: allow guarded source update.
- If install method is `manual-source`: check and advise; do not self-mutate unless a future explicit flag is added.
- If install method is `homebrew`: route to `brew upgrade`.
- If install method is `docker`: route to `docker pull` / `compose pull`.
- If install method is `npm-global`: route to `npm install -g estacoda@latest`.
- If install method is `pnpm-global`: route to `pnpm add -g estacoda@latest` if supported.
- If no stamp detected: detect container runtime (`.dockerenv`, `/run/.containerenv`, `/proc/1/cgroup`). If container detected, route to `docker pull` guidance. Otherwise, print contributor-safe message (fetch + inspect manually).

### Dirty worktree behavior

- managed-source:
  - clean worktree required by default
  - dirty worktree refuses with exit 3
  - `--yes` may auto-stash, proceed, and restore stash on failure
- manual-source:
  - check and advise only
  - no self-mutation
  - no auto-stash
  - no pull/build/update apply

### Expected-branch behavior

- managed-source:
  - update uses expected branch from `.install-method.json`
  - if current branch differs from expected branch, refuse by default
  - `--yes` may switch only if the managed-source stamp proves installer ownership
- manual-source:
  - never auto-switch branches
  - check and advise only

### Update rollback behavior

- Before any mutation: capture current HEAD SHA.
- Before pull: create backup of user state via `backupState()`.
- Before pull: create git stash only if `--yes` was passed and worktree is dirty.
- After pull: run `pnpm install && pnpm run build`.
- If build fails in managed-source:
  - automatically roll back to the captured pre-pull SHA
  - restore stash if one was created
  - preserve user-state backup
  - exit non-zero with recovery instructions
- If install method is manual-source:
  - no source mutation occurs, so automatic rollback should not be needed
- If build succeeds but smoke fails: same rollback path as build failure.
- Gateway mode: log all steps to `~/.estacoda/logs/update.log`; never assume interactive terminal.

## 7. Homebrew Launch Plan

### Tap repo

- **Repo:** `KemetResearch/homebrew-tap` (new external repo)
- **Formula:** `Formula/estacoda.rb`

### Install command

```bash
brew install kemetresearch/tap/estacoda
```

### Artifact/source strategy

**Options:** build from source (git tarball + pnpm) or pull npm-pack tarball.

Building from source matches npm/package.json engine requirements but requires Homebrew to install pnpm. Pulling the npm-pack tarball is simpler and is now a first-class option since PR-I1 makes npm publish ready.

```ruby
url "https://github.com/KemetResearch/EstaCoda/archive/refs/tags/v0.1.0.tar.gz"
# or npm-pack tarball from registry
```

NPM should also be considered as a possible Homebrew formula source strategy; the formula can choose source-build or npm artifact later.

### Checksum strategy

- Use `sha256` of release tarball or npm pack output.
- Update checksum automatically in CI when release is tagged.

### Version bump process

1. Tag `vX.Y.Z` in `KemetResearch/EstaCoda`.
2. CI in EstaCoda repo publishes release tarball and/or npm pack.
3. CI dispatches workflow in `homebrew-tap` to update formula URL + checksum.
4. `homebrew-tap` PR is auto-merged or manually reviewed.

### brew upgrade behavior

```bash
brew upgrade kemetresearch/tap/estacoda
```

- `estacoda update` detects Homebrew install and prints this command.

### brew uninstall behavior

```bash
brew uninstall estacoda
brew untap kemetresearch/tap
```

### Validation commands

```bash
brew install --build-from-source kemetresearch/tap/estacoda
estacoda --version
estacoda --help
brew test estacoda  # if test block added
```

## 8. Docker Launch Plan

### Dockerfile strategy

```dockerfile
FROM node:22.18.0-slim
WORKDIR /app
COPY package.json pnpm-lock.yaml pnpm-workspace.yaml ./
RUN corepack enable && pnpm install --frozen-lockfile
COPY . .
RUN pnpm run build
ENV NODE_ENV=production
ENTRYPOINT ["node", "dist/index.js"]
```

## 10. Risks and Open Questions

1. **NixOS support:** Not a v0.1.0 launch requirement. Can be added post-launch.

2. **`website/` inclusion:** The `files` array correctly excludes `website/`. Confirm this is intentional for package size.

3. **Setup script wrapper strategy:** Should `setup-estacoda.sh` symlink `scripts/estacoda-wrapper.sh` directly, or create a simpler wrapper? `estacoda-wrapper.sh` is already validated and referenced in `verify-package-bin.sh`.

4. **Homebrew formula source strategy:** Build from source (git tarball + pnpm) or pull npm artifact? Building from source is more Hermes-like but requires Homebrew to install pnpm. Pulling npm artifact is simpler and is now a first-class option since PR-I1 makes npm publish ready. The formula can choose later.

5. **Docker image build strategy:** Build from source in Dockerfile (as planned) or copy pre-built `dist/`? Building in Dockerfile is more reproducible and CI-integrated.

6. **Update cache path:** Currently `~/.estacoda/update-cache.json` (global). Should it be profile-local? Global is simpler and update is a global concern.

7. **Install method stamp:** Should the stamp be global (`~/.estacoda/install-method.json`) or repo-local (`.install-method.json` in clone dir)? Repo-local is better for identifying managed dirs.

8. **Managed install directory ownership:** Proven by the presence of `.install-method.json` with matching source URL and expected branch. If missing, treat as manual/contributor.

9. **Fork/upstream behavior for v0.1.0:** Not a launch requirement. If a user forks the repo and installs from their fork, the stamp will show their fork URL. Update behavior should probably warn: "Upstream appears to be a fork. Update will pull from origin. Continue?"

10. **Skill sync on update:** Current skills storage is profile-local under `~/.estacoda/profiles/<id>/skills/`. Bundled skills are in `package skills/`. After a source update, bundled skills are refreshed automatically (they're in the repo). Profile-local skills are user state and must not be touched. Skill sync may be a no-op for v0.1.0.

11. **pnpm/Corepack in Homebrew:** Homebrew formulas can install Node which includes Corepack. The formula would need to run `corepack enable` during build. This is doable but needs testing.

12. **Container detection heuristics:** Detection through `/.dockerenv`, `/run/.containerenv`, and `/proc/1/cgroup` is best-effort. Some container runtimes may not leave these traces. False negatives are acceptable (falls back to manual-source behavior). False positives (detecting container when not in one) should be rare if all three heuristics are checked.

13. **Windows native support:** Not a v0.1.0 launch requirement. WSL2 remains best-effort for v0.1.0. Native Windows would need PowerShell installer, .cmd wrapper, Windows service adapter, file-lock guards, and Windows CI. Estimated +2-3 weeks to v0.1.0.

14. **Fork/upstream sync:** Not a v0.1.0 launch requirement.

## Current-state audit summary

The repo has a skeletal installer (`scripts/install.sh`), a wrapper script, package metadata with `private: true` blocking npm publish, and a minimal update engine that only supports binary artifact updates. It is missing:
- A standalone curl|bash installer
- A git-clone setup script
- npm publish readiness (`private: false`, `repository` field, `publishConfig`)
- Homebrew tap/formula
- Dockerfile and GHCR workflow
- Install method detection (managed-source, manual-source, homebrew, docker, npm-global, pnpm-global, container)
- Source/git update apply (the primary update path)
- Startup background update check
- Complete update command flags (`--check`, `--yes`, `--backup`, `--no-backup`, `--gateway`)
- State preservation that covers repo rollback
- Update resilience (SIGHUP protection, broken-pipe tolerance, update logging)
- Gateway update mode and service restart handoff

## Highest-risk gaps

1. **No source/git update apply path.** The current `update-engine.ts` is binary-only. This is the core Hermes-style behavior required by the release scope.
2. **No install method detection.** Without this, `estacoda update` cannot safely route Homebrew/Docker/npm users or prove managed-install ownership.
3. **No standalone curl|bash installer.** The existing `install.sh` only works when the repo already exists.
4. **`private: true` in package.json.** Blocks npm publish, which is now a v0.1.0 launch path.
5. **No update resilience.** SIGHUP or broken pipe during update may leave the terminal session and repo in an undefined state.
6. **No gateway update mode.** Managed gateway deployments cannot self-update and restart safely.
