# EstaCoda v0.1.0 Release Scope

## Status

Draft release-scope document for the public `v0.1.0` release.

This file defines what EstaCoda intends to claim, support, document, package, and validate for the `v0.1.0` launch.

The information in this file separates:

```text
Current code state      What the code supports today.
Release target          What must be true before v0.1.0 ships.
Public claim            What the README, docs site, release notes, and website may say.
Open decision           A release decision that still needs to be locked.
```

Rule:

```text
If public docs and this file disagree, this file is the release-scope source of truth until superseded by a later release-scope update.
```

## Release Posture

EstaCoda `v0.1.0` is not a small release. It is the first public release that should feel coherent, installable, updateable, documented, and credible.

The release should prioritize:

```text

Hermes-aligned update behavior
clear provider/channel maturity labels

Docusaurus documentation site
fresh-user validation
safe user-state handling
```

The release should not claim broad maturity where the code is only registered, experimental, or not live-proven.

## Supported Operating Systems

### Current code state

The current codebase has no explicit OS support matrix. Existing install behavior is inferred primarily from POSIX shell scripts and Node/pnpm runtime support.

### Release target

For `v0.1.0`, public support is:

```text
macOS 11 Big Sur and later
Modern Linux distributions with systemd and glibc
Docker-capable environments
WSL2 best-effort
Termux best-effort
```

Native Windows is not part of the `v0.1.0` support surface.

### macOS support

```text
Supported floor: macOS 11 Big Sur
Upper bound: none enforced
Reason: Node.js 22.18.0 runtime contract
```

### Linux support

```text
Supported assumption: modern Linux distribution with systemd and glibc
Validated targets: Ubuntu 22.04+ and Debian 12+
Root install behavior: FHS root layout is supported for root installs
```

### WSL support

```text
WSL2 is best-effort.
The Node/pnpm stack runs there and it is a contributor target.
Voice/microphone paths and systemd user services have known edge cases.
```

### Termux support

```text
Termux is best-effort.
The installer resolves a Termux layout using $PREFIX/bin.
Termux is not a primary validation target.
```

### Public claim

```text
EstaCoda v0.1.0 supports macOS 11+, modern Linux distributions with systemd and glibc, and Docker-based usage. WSL2 and Termux are best-effort. Native Windows installation is not part of the v0.1.0 support surface.
```

## Update Scope

### Current code state

The current updater is not yet sufficient for the `v0.1.0` launch target. The present artifact-apply path requires a preexisting update artifact and does not fully implement Hermes-style source update behavior.

### Release target

`v0.1.0` should support Hermes-style update behavior as closely as practical in Node/TypeScript.

Required update capabilities:

```text
install method detection
version resolution
update check
source/git update apply
git fetch
git pull --ff-only
managed clone reconciliation where safe
pnpm dependency reinstall
build after update
post-pull validation
rollback on invalid pulled code
pre-update backup or snapshot
update cache
skill sync where applicable
managed install routing
Homebrew update guidance/delegation
Docker update guidance/delegation
resilience for broken pipes / SIGHUP / gateway mode if stable
background update check if stable
```

### Update commands

Release-target command behavior:

```text
estacoda update --check     check only
estacoda update             apply update with Hermes-style safe default behavior
estacoda update --yes       apply without interactive confirmation where appropriate
estacoda update --backup    create backup before applying
estacoda update --no-backup skip backup where allowed
estacoda update --gateway   non-interactive gateway/service update mode with logging and restart handoff where service manager support exists
```

Startup update checks are enabled by default for `v0.1.0`:

```text
non-blocking background prefetch after session init
6-hour cache
silent failure behavior
startup hint only when update information is available
```

### Managed install routing

```text
managed-source install     estacoda update (guarded source update)
manual-source install      check and advise; no self-mutation
Homebrew install           route to brew upgrade kemetresearch/tap/estacoda or equivalent guidance
Docker install             route to docker pull ghcr.io/kemetresearch/estacoda:v0.1.0 / :latest or compose pull guidance
npm-global install         route to npm install -g estacoda@latest
pnpm-global install        route to pnpm add -g estacoda@latest where supported
manual contributor repo    contributor-safe update behavior; no silent destructive reset
```

npm-global installs are managed by npm. `estacoda update` routes npm-global installs to `npm install -g estacoda@latest`. pnpm-global installs route to `pnpm add -g estacoda@latest` where supported.

### Safety rule

```text
User state must not be destroyed by update.
Contributor worktrees must not be silently destroyed by update.
Hard reset behavior is not allowed outside explicitly managed install directories.
The update implementation must guard against the earlier spec draft's unsafe git reset --hard fallback outside managed install directories.
```

### Public claim

```text
EstaCoda supports source/git self-update for v0.1.0. Homebrew, Docker, and npm installs are updated through their respective package/container managers. EstaCoda protects user state and validates pulled code before finalizing source updates.
```

```text
SHARED.md, USER.md, and MEMORY.md are included as trusted learned memory blocks.
```

Promotion behavior:

```text
User preferences and project facts are recorded in promotions.json.
Existing records can be strengthened by repeated observations.
User preference conflicts can supersede older active preferences.
Preferences can be forgotten/deactivated.
```

External memory:

```text
Disabled by default.
Enabled only when externalMemory.enabled === true and a provider is configured.
Only built-in external provider implemented by code is file.
```

```text
Experimental-only for v0.1.0.
Disabled unless compression.enabled === true and compression.experimental === true.
```

## Security Scope

## Supported Security Modes

### Current code state

Config input aliases normalize to these modes:

```text
strict
adaptive
open
```

Aliases:

```text
manual -> strict
smart  -> adaptive
off    -> open
```

Default mode:

```text
adaptive
```

Strict mode:

```text
Hard blocks deny.
Non-host isolated destructive-command bypass can allow only when no hardline risk remains.
Otherwise follows capability-first defaults.
```

Adaptive mode:

```text
Hard blocks deny.
credential-access, sandbox-escape, and spend-money deny deterministically.
Non-host isolated destructive-command bypass may allow.
Non-destructive risk classes follow capability-first defaults.
Ambiguous destructive local actions ask unless a known benign false-positive rule matches.
Optional smart assessor can classify ask cases via auxiliary model route assessor.
```

Open mode:

```text
Allows actions by default unless they match the hard dangerous-command floor.
Still denies hard blocks.
```

Capability-first defaults:

```text
read-only-local allowed only in trusted workspace
read-only-network allowed
workspace-write allowed only in trusted workspace
shared-state-mutation allowed only in trusted workspace
external-side-effect allowed only for active target conversation
credential access asks by default
destructive local actions ask by default
spending money asks by default
sandbox escape asks by default
browser.cdp asks by default
```

Supported environment types:

```text
host
docker
singularity
modal
daytona
vercel_sandbox
```

Default environment type:

```text
host
```

### Public claim

```text
EstaCoda v0.1.0 supports strict, adaptive, and open security modes. Adaptive is the default. Hard safety blocks remain active even when open/yolo-style behavior is enabled.
```

## Known Limitations

The following limitations are in scope for the current code state and must either be fixed before `v0.1.0` or documented clearly if they remain true at release.

```text
The code has no explicit OS support matrix; install support is inferred from a POSIX shell script.
Runtime state does not use ESTACODA_HOME as the state root, even though the installer uses it for binary placement.
Release artifact apply requires a preexisting ESTACODA_UPDATE_ARTIFACT; the updater does not download and apply artifacts itself.
Cloud browser providers are registered but cannot create sessions.
Web search/crawl providers are registered but not implemented as live providers.
Anthropic, MiniMax, and Nous are catalog-known but not runnable LLM providers in this build.
WhatsApp is gated experimental and not live-proven.
Discord and email are present but not live-proven.
Email attachments are not supported by the capability registry.
Discord attachments, threads, and progress streaming are not supported by the capability registry.
Session compression is experimental-gated and disabled unless both enabled and experimental flags are true.
External memory only has a built-in file provider implementation.
Skill learning can create local skills only for detected bounded workflows.
Authority-expanding skill mutations are refused by the mutation path.
Browser private URL access is disabled by default.
Raw browser.cdp is supported but approval-gated by default.
TaskFlow integration is best-effort and only wires when the session DB is SQLite.
```

```text
skill files without frontmatter

private/internal browser URL access unless explicitly allowed, with metadata endpoints always blocked
```

## Release Promotion Rules

A feature can move from current/experimental/registered to public `v0.1.0` supported only when all are true:

```text
implementation exists
configuration path exists
docs exist
validation exists
fresh-user behavior is understood
failure mode is documented
security/state implications are documented
```

A provider/channel/tool cannot be called supported merely because it appears in metadata or a registry.

## Required Validation Before Public Claim

Minimum evidence levels:

```text
live-proven       manually validated in realistic usage
smoke-tested      covered by smoke test or equivalent scripted check
test-covered      covered by unit/integration tests
implemented       code exists but not enough evidence for public stability claim
registered        appears in metadata/registry only
unsupported       not usable in current release
```

Public docs should use these labels consistently where maturity matters.
