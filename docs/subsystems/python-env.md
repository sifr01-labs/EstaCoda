# Managed Python Environments

EstaCoda can run capabilities that need Python packages without mutating the system Python installation or relying on whatever happens to be installed on the host.

Managed Python environments are runtime-owned dependency environments. They are not skill state, not profile state, and not user-facing artifacts.

## What this subsystem owns

The Python environment subsystem owns:

- registered capability specifications
- per-capability virtualenv paths
- per-capability pip cache paths
- install and verification locks
- environment manifests
- status, setup, verify, upgrade, and reset operations
- bounded and redacted diagnostics

It does not own:

- profile-local skill state
- temporary research caches
- generated markdown or HTML artifacts
- package lists written by skills
- package lists generated by providers

## Capability specifications

A capability specification is registered by the runtime.

A skill may request a capability by ID. A skill may not define packages, imports, install commands, paths, or versions.

A capability specification contains:

```ts
type ManagedPythonCapabilityEnvSpec = {
  id: string;
  version: string;
  packages: string[];
  verifyImports: string[];
  pythonVersion?: string;
  estimatedInstallSizeMb?: number;
  optionalGroups?: Record<string, {
    packages: string[];
    verifyImports: string[];
    estimatedInstallSizeMb?: number;
  }>;
};
```

The runtime registry is the source of truth for:

- which packages install
- which imports verify
- which optional groups exist
- where the environment lives
- how much disk space is expected

Do not read package lists from `SKILL.md`, provider output, generated requirements files, or arbitrary shell text.

## Paths

Generic capability environments live under the EstaCoda state root:

```text
<stateRoot>/python-envs/<capability-id>/
```

The pip cache for a capability lives under:

```text
<stateRoot>/cache/pip/<capability-id>/
```

The manifest lives at:

```text
<stateRoot>/python-envs/<capability-id>/env.json
```

These paths are not profile-local. A profile may have its own skill state, but dependency environments are runtime capability state.

## Manifest

Each managed capability environment has a manifest.

The manifest records:

```ts
type ManagedPythonCapabilityEnvManifest = {
  id: string;
  version: string;
  specHash: string;
  installedPackages: string[];
  installedGroups: string[];
  pythonPath: string;
  envPath: string;
  createdAt: string;
  updatedAt: string;
  verifiedAt?: string;
  status: "installing" | "installed" | "verified" | "broken";
};
```

`specHash` fingerprints the effective registered spec, including selected optional groups. If the registered spec changes, status can report `upgrade_required` instead of guessing from the filesystem.

Manifest writes must be atomic.

## Optional groups

Optional groups add packages and import checks to the same capability environment.

They do not create a second virtualenv unless a future capability is deliberately registered as a separate capability for isolation.

This shape is valid:

```yaml
pythonCapabilities:
  - id: pdf-extraction
    required: true
    groups: []
  - id: pdf-extraction
    required: false
    groups: ["advancedOcr"]
```

The base capability is required. The `advancedOcr` group is optional. Failure to resolve the optional group must not make the required base capability unavailable.

Registered built-in capability IDs include:

- `faster-whisper`: local STT package support.
- `pdf-extraction`: PyMuPDF/PyMuPDF4LLM extraction, with optional `tables` and `advancedOcr` groups.
- `pdf-editor`: nano-pdf editing support.

## Operator commands

The local operator surface is:

```bash
estacoda python-env list
estacoda python-env status <id>
estacoda python-env setup <id>
estacoda python-env verify <id>
estacoda python-env upgrade <id>
estacoda python-env reset <id>
```

Optional groups can be selected with the supported group flags:

```bash
estacoda python-env setup <id> --group <name>
estacoda python-env setup <id> --groups <a,b>
estacoda python-env status <id> --group <name>
```

`list` and `status` are read-only.

`setup` creates the environment, installs pinned packages from the registered spec, verifies imports, and writes the manifest.

`verify` checks configured imports only. It does not install packages.

`upgrade` compares the installed manifest against the current registered spec. If the spec changed, it installs the current pinned packages, verifies imports, and updates the manifest.

`reset` deletes the generic capability environment path. It is destructive and requires confirmation.

## Runtime resolver

Normal skill execution uses the resolver in non-installing mode.

The resolver returns a Python path only when the requested registered capability and selected groups are already installed and verified.

It must not:

- create a virtualenv
- invoke pip
- run setup
- run upgrade
- run reset
- repair automatically
- install from skill metadata
- install from provider output

Unavailable capabilities return structured status and a repair command for the local operator.

## Skill metadata

Skills may declare Python capabilities like this:

```yaml
pythonCapabilities:
  - id: example-capability
    required: true
    groups: []
```

For example, a registered capability can expose an optional group:

```yaml
pythonCapabilities:
  - id: pdf-extraction
    required: false
    groups: ["advancedOcr"]
```

Each entry may contain only:

```yaml
id: <registered capability id>
required: true | false
groups: []
```

Unknown keys fail validation. This is intentional. Skill metadata must not become a package installation surface.

Rejected examples include:

```yaml
pythonCapabilities:
  - id: example
    required: true
    groups: []
    packages:
      - httpx
```

```yaml
pythonCapabilities:
  - id: example
    required: true
    groups: []
    installCommand: pip install httpx
```

```yaml
pythonCapabilities:
  - id: example
    required: true
    groups: []
    pythonPath: /tmp/python
```

## faster-whisper compatibility

The existing faster-whisper local STT environment keeps its legacy path behavior.

Generic capability environments use:

```text
<stateRoot>/python-envs/<capability-id>/
```

The legacy faster-whisper path remains separate unless a future migration explicitly changes it.

## Safety rules

The subsystem follows these rules:

- install exact pinned packages only
- install only from runtime-registered capability specs
- require explicit local approval before network package installation
- keep pip cache under EstaCoda state
- never mutate system Python
- never mutate user-owned virtualenvs
- never install during normal skill execution
- never auto-install from gateway-triggered execution
- never install from provider-generated shell text
- bound and redact diagnostics before printing
- treat reset as destructive local operator action
