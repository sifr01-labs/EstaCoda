# بيئة التشغيل

بيئة تشغيل EstaCoda هي البيئة التنفيذية لدور واحد في الجلسة. تُنشأ من جديد لكل دور في وضع البوابة وتستمر عبر الأدوار في وضع CLI التفاعلي. كل ما تحتاجه بيئة التشغيل — الإعداد، والاعتمادات، والمهارات، والذاكرة، والأدوات — يُحمل عند الإنشاء من ملف شخصي واحد مُختار.

تشرح هذه الصفحة كيفية إنشاء بيئة التشغيل، وما تحله، وأين يمكن أن تفشل.

---

## ما هي بيئة التشغيل

بيئة التشغيل هي مجموعة من الأنظمة الفرعية المهيأة التي تتعاون لمعالجة دور مستخدم واحد:

- إعداد ملف شخصي واحد مُختار
- سجل مزود واحد مع مسارات رئيسية وإضافية محلولة
- سجل أدوات واحد مع أدوات مدمجة وأدوات مكتشفة عبر MCP
- سجل مهارات واحد مع مهارات مرئية لهذه الجلسة
- سياسة أمان ووحدة تحكم بالموافقة واحدة
- اتصال قاعدة بيانات جلسات واحد
- مخزن ذاكرة ومزود واحد
- حلقة وكيل واحدة تنسق بينهم جميعًا

بيئة التشغيل غير مشتركة عبر الملفات الشخصية. تغيير الملف الشخصي يعني إنشاء بيئة تشغيل جديدة بإعداد واعتمادات وذاكرة مختلفة.

---

## اختيار الملف الشخصي

يُحمل إعداد بيئة التشغيل من ملف شخصي واحد مُختار بالضبط:

- `profileId` صريح يُمرر إلى الأمر
- الملف الشخصي النشط إذا لم يُعطَ معرف صريح
- `default` إذا لم يُعيّن ملف شخصي نشط

لا يوجد دمج إعداد مستخدم/مشروع. ثقة مساحة العمل هي مدخل تحكم سلوكي للإجراءات المحلية وبدء MCP؛ لا تغيّر ملف الإعداد الذي يُحمل.

حالة الملف الشخصي تعيش تحت `~/.estacoda/profiles/<id>/`. راجع [الهندسة المعمارية](./architecture.md) لخريطة حدود الحالة الكاملة.

---

## بناء بيئة التشغيل

`createRuntime()` تبني الأنظمة الفرعية بترتيب ثابت:

1. **مخازن الثقة والموافقة** — `WorkspaceTrustStore`، `WorkspaceApprovalController`
2. **بنية المزود** — `ProviderRegistry`، حل المسار الإضافي للنموذج
3. **واجهة المتصفح الخلفية** — mock أو CDP حقيقي
4. **موسع السياق ومحمل المشروع** — `ContextReferenceExpander`، `ProjectContextLoader`
5. **الذاكرة** — `MemoryStore`، `LocalMemoryProvider`
6. **الاستمرارية** — `CronStore`، `SessionDB`، `ArtifactStore`
7. **مدير العمليات**
8. **بنية الأدوات** — `ToolRegistry`، `ToolExecutor`، `ToolCallPlanner`
9. **بنية المهارات** — `SkillRegistry`، `SkillLearningManager`، `SkillEvolutionStore`
10. **مدير التفويض**
11. **منفذ المزود**
12. **مكونات وقت التشغيل المستخرجة** — `RunRecorder`، `ToolPlanRunner`، `ProviderTurnLoop`، `SkillWorkflowExecutor`، `NativeToolExecutor`، `RuntimeRouter`
13. **AgentLoop** — دورة التنسيق

أي تغيير في توقيع الباني ينتشر عبر هذا الملف. لا يوجد حاوية حقن تبعية أو حدود إضافية.

## حل مسار المزود

تحل بيئة التشغيل ثلاثة أنواع من مسارات المزود:

### المسار الرئيسي

يُعرّف تحت `model` في إعداد الملف الشخصي. يُستخدم للاستدلال العادي. إذا أصبح المسار الرئيسي غير قابل للتشغيل — مفتاح API مفقود، أو اسم نموذج قديم، أو فشل المزود — تُبلغ EstaCoda عن الفشل. لا ترجع صامتةً إلا إذا كانت مسارات احتياطية صريحة مُهيأة.

### مسارات الاحتياطي

قائمة مرتبة تحت `model.fallbacks`. تُجرّب فقط عندما يفشل المسار الرئيسي في وقت التنفيذ. تحافظ على `apiKeyEnv` و `baseUrl` و `apiMode` و `authMethod` عند توفرها. إذا فشلت جميع الاحتياطات، يُبلغ الدور عن الخطأ ويتوقف.

### المسارات الإضافية

مسارات متخصصة لمهام مثل الرؤية والضغط وتقييم الأمان واستخراج الويب والبحث في الجلسات. تُهيأ تحت `auxiliaryModels`. تحل عبر نفس بنية المزود كالمسار الرئيسي.

| المسار | الغرض | النضج |
|---|---|---|
| `vision` | تحليل الصور | `implemented` |
| `compression` | الضغط الدلالي للجلسات | `experimental` |
| `assessor` | تصنيف الموافقة الذكية | `implemented` |
| `web_extract` | استخراج الويب | `implemented` |
| `session_search` | البحث الدلالي في الجلسات | `implemented` |
| `mcp` | تفويض أدوات MCP | `implemented` |
| `memory_flush` | عمليات الذاكرة | `implemented` |
| `delegation` | تفويض الوكيل الفرعي | `implemented` |
| `skills_library` | توزيع المهارات | `implemented` |
| `title_generation` | توليد عناوين الجلسات | `implemented` |
| `curator` | التنظيم المنطقي للذاكرة | `implemented` |
| `memory_compaction` | ضغط ملفات الذاكرة | `implemented` |
| `profile_context` | توليد سياق الملف الشخصي | `implemented` |

أسماء المسارات الإضافية غير المعتمدة ترمي خطأ أثناء تطبيع الإعداد.

## سجل الأدوات وحد التنفيذ

`ToolRegistry` يسجل الأدوات المدمجة عند تحميل الوحدة والأدوات المكتشفة عبر MCP عند إنشاء بيئة التشغيل. `ToolExecutor` ينفذ إجراءات الأداة الملموسة تحت سياسة الأمان النشطة.

**قاعدة الحد:** المزود لا يعرف عن الأدوات. منفذ الأدوات لا يعرف عن المزودين. فقط حلقة الوكيل تربط بينهما.

`ToolPlanRunner` يحول استدعاءات أدوات المزود إلى خطط قابلة للتنفيذ، ويدير التزامن الآمن للأدوات، ويفرض سقوف الفشل، ويبني حزم النتائج للاستمرارية. كان مضمنًا سابقًا في `AgentLoop`؛ الآن مكون مستخرج.

فئات مخاطر الأدوات: `safe`، `caution`، `external-side-effect`، `irreversible`. سياسة الأمان تتحكم عبر `targetKey`، لا عبر ملخص العرض.

## حدود البوابة والجلسة وذاكرة التخزين المؤقت لوقت التشغيل

### وضع البوابة

في وضع البوابة، تُنشأ بيئة تشغيل جديدة لكل دور وارد. هذا يضمن:

- تغييرات الإعداد تسري فورًا
- تجاوزات النموذج المحدودة بالجلسة تُعاد التحقق منها
- ذاكرات التخزين المؤقت لوقت التشغيل لا تتسرب بين المستخدمين

المقايضة هي تكلفة الإنشاء. إنشاء بيئة تشغيل البوابة مُحسّن لكنه ليس مجانيًا.

### وضع CLI

في وضع CLI التفاعلي، تستمر بيئة التشغيل عبر الأدوار ضمن جلسة. هذا يقلل من تكلفة الإنشاء ويحافظ على حالة الجلسة في الذاكرة.

### قاعدة بيانات الجلسة

الجلسات مُخزّنة في `~/.estacoda/sessions.sqlite`، مُنطقة بـ `profile_id`. قاعدة بيانات الجلسة تفوق أي مثيل واحد لبيئة التشغيل. سياق جلسة القناة يُحفظ تحت حالة بوابة الملف الشخصي المرتبط.

### ذاكرة التخزين المؤقت لوقت التشغيل

طبقات ذاكرة تخزين الموجه (الهوية، الأمان، سياق المشروع، موارد المهارات) تُعاد بناؤها فقط عند تغيير البيانات الأساسية. تاريخ الجلسة والسياق الحي للمستخدم يُعاد بناؤهما كل دور.

---

## تجميع موجه الذاكرة

يُعد سياق الذاكرة لكل دور بواسطة `MemoryRecallOrchestrator` ويُصيّر بواسطة `MemoryPromptContextBuilder`. خط تجميع الموجه يتضمن:

1. السياق الأساسي لموجه الذاكرة
2. سياق المشروع (بما في ذلك `AGENTS.md`)
3. إشعار الضغط الاختياري
4. تاريخ الجلسة
5. الاستذكار الاختياري للجلسة والاستذكار الخارجي
6. الرسالة الحية للمستخدم والمرفقات
7. النية وتعليمات المهارة وإعداد المهارة ومواردها
8. خطة سير العمل
9. قائمة الأدوات
10. السياق المرجعي الصريح
11. نتائج الأدوات / ملاحظات الاستمرارية

الاستذكار والاستذكار الخارجي وملخصات الضغط هي سياق مرجعي فقط. تُدرج للاستمرارية لكن لا تُعامل كتعليمات صاحبة صلاحية.

الضغط الدلالي لدور المزود يملكه `AgentLoop`، لا `ProviderTurnLoop`. عند التمكين وتجاوز العتبة، يحافظ `AgentLoop` على النص الأصلي للأب بالضغط في جلسة فرعية قبل تجميع موجه المزود.

## أعطال بدء تشغيل بيئة التشغيل

**إعداد الملف الشخصي مفقود:** `createRuntime` يفشل مبكرًا إذا لم يكن دليل الملف الشخصي المختار أو `config.json` موجودًا. CLI يُبلغ عن الملف الشخصي المفقود ويقترح `estacoda profile switch` أو الإعداد.

**المسار الرئيسي غير قابل للتشغيل:** مفتاح API مفقود، أو اسم نموذج غير صالح، أو فشل المزود. `estacoda model diagnose` يُبلغ عن الفشل بالضبط. `estacoda model setup` يُصلح المسار.

**المسار الإضافي مفقود:** إعداد مسار إضافي مفقود أو مشوه. النظام الفرعي المستدعي يرتدع حسبما وثّق (مثلاً الموافقة اليدوية إذا فشل `assessor`).

**خادم MCP غير متاح:** أدوات MCP غير مسجّلة. بيئة التشغيل تستمر بدون أدوات MCP. تحقق من `estacoda gateway diagnose` لجاهزية MCP.

**واجهة المتصفح الخلفية غير متاحة:** إذا كان `browser.backend` هو `local-cdp` وChrome لا يعمل على `cdpUrl` المُعيّن، تُبلغ عمليات المتصفح عن أخطاء الاتصال. فعّل `autoLaunch` أو ابدأ Chrome يدويًا.

**قفل SQLite أو تلف:** عمليات الجلسة تفشل. `better-sqlite3` يستخدم دلالات SQLite المتزامنة. تحقق من أذونات الملف على `~/.estacoda/sessions.sqlite`.

## كيفية فحص حالة بيئة التشغيل

```bash
# المسار الرئيسي الحالي والجاهزية
estacoda model show

# تشخيص حي ضد المزود المُهيأ
estacoda model diagnose

# قائمة المزودين المعروفين في الكاتالوغ
estacoda model list

# جاهزية البوابة (تتضمن MCP، القنوات، cron)
estacoda gateway diagnose

# حالة البوابة الكاملة
estacoda gateway status

# الجلسات الأخيرة
estacoda sessions list

# الجلسة الحالية
estacoda sessions current
```
