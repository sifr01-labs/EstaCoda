# أوامر الشرطة المائلة

أوامر الشرطة المائلة هي أسطح تحكم متاحة داخل جلسة EstaCoda نشطة. تفحص الحالة، وتبدّل النماذج، وتدير الموافقات، وتتابع الجلسات بدون مغادرة المحادثة. بعض أوامر الشرطة المائلة تمتلك نظيراً CLI مكافئاً؛ الاختلافات في السلوك مذكورة حيث توجد.

ليست جميع أوامر الشرطة المائلة متاحة في كل السياقات. جلسات البوابة البعيدة وجلسات CLI المحلية قد تُظهر فيهما مجموعات مختلفة. الأوامر التي تتطلب TTY أو منتقي تفاعلي تتدرج بشكل ملائم عند التشغيل غير التفاعلي.

---

## طريقة الاستخدام

اكتب أمر الشرطة المائلة في محث الجلسة:

```bash
/status
/model set openai/gpt-4o
/approvals
/exit
```

الشرطة الفارغة `/` تفتح قائمة الشرطات المائلة.

## التحكم في الجلسة

### `/sessions` أو `/session`

يعرض الجلسات الأخيرة للملف التعريف النشط.

**الحالة المُعدّلة:** لا شيء (قراءة فقط).

**الاختلاف في البوابة:** امر `/sessions` في البوابة يعرض الجلسات الأخيرة لمستخدم القناة.

### `/switch <session-id>`

يبدّل التشغيل الحالي إلى جلسة موجودة.

**الحالة المُعدّلة:** قاعدة بيانات جلسات SQLite (مؤشر الجلسة النشط).

**أنماط الفشل:**
- يرفض إذا لم تُوجد الجلسة في الملف التعريف النشط.
- يرفض إذا لم يكن `switchRuntime` متاحاً في السياق الحالي.

### `/reset` أو `/new`

يبدأ جلسة جديدة ويحدّث لقطة المهارات/الإعداد.

**الحالة المُعدّلة:** ينشئ سجل جلسة جديداً؛ الجلسة القديمة تبقى في قاعدة البيانات.

**أنماط الفشل:** يرفض إذا لم يكن `refreshRuntime` متاحاً.

### `/search <query>` أو `/find <query>`

البحث في تاريخ الجلسة.

**الحالة المُعدّلة:** لا شيء (قراءة فقط).

### `/compact [topic]`

يضغط السياق الحالي داخل الجلسة عبر الضغط الدلالي.

**الحالة المُعدّلة:** قاعدة بيانات جلسات SQLite (كتابة حدث ضغط).

**السلوك:** غير دوّار في تطبيق CLI؛ لا يتبنى جلسة فرعية مضغوطة. أمر `/compact` في البوابة له سلوك تبنية منفصلة ويستطيع الاحتفاظ بالمحادثة الصادرة عبر التبدّل إلى جلسة فرعية مضغوطة.

**أنماط الفشل:** إذا فشل الضغط، السياق الأصلي محفوظ.

## تبديل النموذج

### `/model`

يعرض النموذج النشط والمزود دون تغيير الإعداد المحفوظ.

**الحالة المُعدّلة:** لا شيء (قراءة فقط).

### `/model <provider>/<model>` أو `/model set <provider>/<model>`

يعيّن تجاوز نموذج مرتبط بالجلسة.

**الحالة المُعدّلة:** قاعدة بيانات جلسات SQLite (تجاوز النموذج للجلسة).

**السلوك:**
- التجاوز مرتبط بالجلسة النشطة بشكل افتراضي. التجاوز يدوم مع الجلسة ويُعاد التحقق منه عند إنشاء التشغيل.
- التجاوزات القديمة أو الغير صالحة تتم تجاهلها بشكل غير قاتل؛ يعوّد التشغيل إلى المسار الأساسي المُهيّأ.
- مسارات الاحتياطي والمسارات المساعدة تبقى محفوظة عند التبدّل بين الجلسات.
- المنتقي يعرض فقط الخيارات الجاهزة والقابلة للتشغيل. المزودون الفاقدون بيانات الاعتماد يرفضون مع إرشادات إعداد طرفية.

### `/model --global <provider>/<model>` أو `/model set --global <provider>/<model>`

حفظ المسار المختار كمسار أساسي للملف التعريف.

**الحالة المُعدّلة:**
- `~/.estacoda/profiles/<id>/config.json`
- قاعدة بيانات جلسات SQLite (مسح تجاوز الجلسة)

**السلوك:** يتطلب عملية مشتملة موثوقة. لا يجمع بيانات الاعتماد مباشرة. يرفض إذا كان مساحة العمل غير موثوقة.

### `/model clear`

يمسح تجاوز النموذج المرتبط بالجلسة.

**الحالة المُعدّلة:** قاعدة بيانات جلسات SQLite.

**أنماط الفشل:** `/model --global clear` مرفوض. مسح المسار الأساسي ليس له معنى معرّف في المنتج.

**الحالة المُعدّلة:** `~/.estacoda/profiles/<id>/trust.json`.

يسحب ثقة مساحة العمل.

**الحالة المُعدّلة:** `~/.estacoda/profiles/<id>/trust.json`.

### `/workspace.trust.status`

يعرض حالة ثقة مساحة العمل الحالية.

**الحالة المُعدّلة:** لا شيء (قراءة فقط).

## الأمن والموافقات

### `/approvals`

يعرض الموافقات الحالية: مرة واحدة، للجلسة، والمستمرة.

**الحالة المُعدّلة:** لا شيء (قراءة فقط).

### `/revoke <approval-id>`

يسحب موافقة مستمرة بواسطة المعرف.

**الحالة المُعدّلة:** مخزن موافقات التشغيل.

**أنماط الفشل:** يرفض إذا لم يكن `revokeApproval` متاحاً في التشغيل الحالي.

### `/security` أو `/security debug`

يفحص القرارات الأمنية الأخيرة. `debug` يضمّن بيانات المراجعة التفصيلية.

**الحالة المُعدّلة:** لا شيء (قراءة فقط).

### `/yolo`

يبدّل وضع الموافقة المفتوح/YOLO للجلسة.

**الحالة المُعدّلة:** حالة التشغيل.

**السلوك:** يوافق تلقائياً على الأفعال المؤهلة في الجلسة الحالية. كتل الأمان الصلبة تبقى سارية. التفاؤل ليس استراتيجية للمزود.

## Cron

### `/cron <subcommand>`

إدارة المهام المجدولة من داخل الجلسة.

```bash
/cron list
/cron add --schedule <schedule> --command "<prompt>"
/cron edit <job-id> [flags]
/cron show <job-id>
/cron history [job-id]
/cron run <job-id>
/cron pause <job-id>
/cron resume <job-id>
/cron remove <job-id>
/cron tick
```

**الحالة المُعدّلة:**
- `~/.estacoda/profiles/<id>/cron/jobs.json`
- `~/.estacoda/sessions.sqlite`

**السلوك:** يستخدم CronStore الملف التعريف النشط. الدورة اليدوية تشغّل المهام المستحقة فوراً داخل runtimes معزولة للـ cron. يدعم cron add/edit عناصر التحكم المتقدمة نفسها في CLI، ومنها skills وscripts ووضع no-agent وسلاسل contextFrom وتحكمات model/tool و`--workdir` الموثوق والمحصور.

راجع [المهام المجدولة](../user-guide/cron.md) للتفاصيل.

## المتصفح

### `/browser <subcommand>`

إدارة اتصال المتصفح المحلي/CDP.

```bash
/browser status
/browser connect
/browser connect http://127.0.0.1:9222
/browser disconnect
```

**الحالة المُعدّلة:** حالة المتصفح في التشغيل.

**السلوك:** هذا الأمر slash هو سطح التحكم المحلي داخل الجلسة لاتصال CDP. يمكنه فحص الحالة، أو ضبط endpoint محلي لـ CDP، أو تحديث runtime عندما يتوفر ذلك، أو فصل backend المتصفح للملف التعريفي النشط.

**ملاحظة المتصفح السحابي:** إعداد Browserbase وموافقة الإنفاق أوامر CLI وليست أوامر slash: استخدم `estacoda browser setup --backend browserbase --cloud-provider browserbase --hybrid-routing` و`estacoda browser approve-cloud` و`estacoda browser revoke-cloud`.

**أنماط الفشل:** يتدهور إذا لم يكن هناك backend متصفح مُهيّأ. لا تُدار موافقة إنفاق Browserbase عبر أوامر slash.

## Workflow

يتطلب قاعدة بيانات جلسات SQLite. متاح عندما يكون Workflow مربوطاً.

```bash
/workflow begin <objective>
/workflow begin --skill <skillName> <objective>
/workflow status [runId]
/workflow pause <runId> [reason]
/workflow resume <runId>
/workflow interrupt <runId> [reason]
/workflow cancel <runId> [reason]
/workflow steer <runId> <guidance>
/workflow approve <stepId>
/workflow reject <stepId> [reason]
/workflow retry <stepId>
/workflow skip <stepId> [reason]
/workflow checkpoint <runId> <name>
/workflow trace [runId] [limit]
/workflow summarize <runId>
/workflow activate <runId>
/workflow deactivate
```

إذا تم حذف `runId` في `status` و `trace`، يستخدم تشغيل Workflow النشط.

**الحالة المُعدّلة:** قاعدة بيانات جلسات SQLite (جداول `workflow_events`، `workflow_steps`).

**السلوك:**
- `/workflow begin <objective>` ينشئ تشغيل Workflow محافظاً بخطوة واحدة، ويبدأه، ويفعّله في الجلسة التفاعلية الحالية.
- `/workflow begin --skill <skillName> <objective>` يحل اسم المهارة، ويجمع playbook الخاص بها، ويحوله إلى `WorkflowPlan`، ثم ينشئ التشغيل ويبدأه ويفعّله في الجلسة التفاعلية الحالية.
- أمر begin الناجح يطبع `Created workflow: <runId>` و`Started workflow: <runId>` و`Activated workflow: <runId>`.
- `/workflow begin <objective>` بدون `--skill` يسجل provenance صريحاً ولا يستخدم تحويل playbook.
- `/workflow steer` يسجّل حدث `OperatorEvent` غير مستهلك. في الدور التالي للمحوّل، يُضاف التوجيه كبدئة لنص المستخدم. الأحداث تُعلّم مستهلكة وتظهر في `/workflow trace`.
- `/workflow activate` يربط الجلسة الحالية بتشغيل Workflow. `/workflow deactivate` يمسح الربط.

**أنماط الفشل:**
- الهدف المفقود يرجع نص الاستخدام.
- المهارة غير المعروفة ترجع خطأ واضحاً.
- يُرفض التوجيه لتشغيلات Workflow في حالات نهائية.
- إعادة المحاولة تعمل فقط إذا كان `idempotent` أو `safeToRetry` صحيحاً وتحت `maxRetries`.
- التخطي يعمل فقط إذا لم يبدأ الخطوة وكان `allowSkipIfSkippable` صحيحاً.
- Workflow begin لا ينفذ automatic workflow promotion، ولا complex-request detection، ولا سلوك Agent Evolution، ولا إنشاء Workflow تلقائياً من اختيار المهارة العادي داخل AgentLoop. خيار `--skill` اشتراك صريح. `--use-selected-playbook` غير مدعوم.

## النظام

### `/reload-mcp`

يعيّد تحميل إعداد MCP ويحدّث أدوات MCP للجلسة.

**الحالة المُعدّلة:** سجل أدوات التشغيل (يعاد البناء من الإعداد الحالي).

**السلوك:** يتطلب `refreshRuntime`. أمر CLI الفردي يرى إعداد MCP الحالي تلقائياً بدون تحميل يدوي.

### `/doctor`

ينفذ فحصاً صحياً سريعاً داخل الجلسة.

**الحالة المُعدّلة:** لا شيء (قراءة فقط).

### `/clear` أو `/cls`

يمسح الطرفية.

**الحالة المُعدّلة:** لا شيء.

### `/exit` أو `/quit`

ينهي الجلسة.

**الحالة المُعدّلة:** لا شيء (ينهي العملية).

## التوجيهات في نمط الموافقة

عندما يصل تنفيذ أداة CLI إلى نمط موافقة نشط، تُقبل الإجابات التالية:

- `once` — منح هذا الإجراء مرة واحدة وإعادة المحاولة
- `session` — منح الإجراءات المطابقة للجلسة وإعادة المحاولة
- `always` — حفظ موافقة مستمرة للإجراءات المطابقة وإعادة المحاولة
- `deny`، `reject`، `no`، `n` — الرفض بدون إعادة

الألقاب بالشرطة داخل نمط الموافقة نفسه:

- `/approve once`
- `/approve session`
- `/approve always`
- `/deny`

هذه تُعادل إلى نفس الخيارات الأساسية. إدخال غير صالح مثل `/approve banana` يتبع مسار الإرشاد الحالي ولا يمنح الموافقة.
